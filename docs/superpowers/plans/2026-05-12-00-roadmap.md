# STT + QDA Implementation Roadmap

**Spec:** [`docs/superpowers/specs/2026-05-12-stt-qda-design.md`](../specs/2026-05-12-stt-qda-design.md)
**Date:** 2026-05-12

The implementation is split into 8 sub-plans. Each plan produces working, testable software.

**Note on freshness:** Plans 02–08 were drafted upfront and may need refinement when actually executed — the implementer subagent will likely surface details (API drift, library quirks, scaffold layout differences) that the original draft missed. Plan 01 went through this in execution and surfaced the `stt` → `stt_app_lib` lib-name substitution. Treat the plans as opinionated starting points, not contracts.

## Dependency graph

```
01 Foundation
    ↓
02 Database & codebook core
    ↓
03 Transcription pipeline
    ↓
04 Tagging UX
    ↓                           ↘
05 AI features                   06 Import variants
    ↓                           ↙

    ↓
```

05 and 06 are independent and can be done in either order after 04. Everything else is strictly linear.

## Sub-plans

### 01 — Foundation
**File:** `2026-05-12-01-foundation.md` (written)
**Goal:** A Tauri 2 app launches, picks/creates a project folder, opens an empty workspace shell, and exposes a minimal settings page (Gemini API key).
**Deliverables:**
- Tauri 2 scaffold with React + TypeScript + Vite frontend
- TanStack Router with three routes: `/picker`, `/workspace/$projectId/*`, `/settings`
- Jotai for state, atoms defined outside components
- Plain CSS / CSS Modules (no Tailwind, no CSS-in-JS)
- `react-i18next` wired up with `en.json` and `cs.json`, OS-language auto-detection
- Project Picker (open recent, open folder, new project) — calls Rust to create/load `project.sqlite` skeleton (one `project_meta` row, migrations table)
- Empty Workspace shell with three-pane layout placeholders
- Settings page with one field (Gemini API key) persisted to global config
- Migration framework in Rust (versioned, applied on project open)
**Done when:** user can launch the app, create a new project folder, the app stores meta + Gemini API key, and reopens the project from a recent list. Workspace shows three empty panes. No transcript / codebook / tagging yet.

### 02 — Database & codebook core
**File:** `2026-05-12-02-database-codebook.md` (written; refine when executing)
**Goal:** Full SQLite schema in place, plus codebook CRUD (clusters/categories/tags) end-to-end with uniqueness enforcement.
**Deliverables:**
- All tables from spec §4 created via migrations
- Typed query layer in Rust (`db/queries/*`) for every entity
- Tauri commands for codebook CRUD (cluster/category/tag create/rename/delete/reorder)
- Uniqueness violations surface as typed errors, handled gracefully in UI
- Left-pane CodebookTree component (create/rename/delete via inline editing)
- Color picker per entity
- Description field per entity
- Atom subscriptions update tree when codebook changes
- Rust tests cover every uniqueness constraint, cascade, and reorder behavior
**Done when:** user can build a complete codebook through the UI; uniqueness violations are caught at the DB level and shown clearly.

### 03 — Transcription pipeline
**File:** `2026-05-12-03-transcription.md` (written; refine when executing)
**Goal:** Audio import + on-demand transcription via Gemini, with progress events and resumability.
**Deliverables:**
- ffmpeg/ffprobe bundled as Tauri sidecars (Linux/macOS/Windows)
- Rust port of `gemini_main.py`: normalize → chunk → upload → generateContent → validate → save (`src-tauri/src/transcription/*`)
- Retry/backoff, MAX_TOKENS sub-chunk fallback, schema validation, timestamp rescaling — feature parity with Python
- `transcribe_start` / `transcribe_cancel` commands; progress events via Tauri event channel
- "Add interview (audio)" UI: file dialog → copies into `media/` → creates `interview` row
- "Transcribe" button on interview, progress strip in left pane
- Minimal transcript viewer in center pane (read-only segment list with speakers + timestamps)
- `ai_run` rows written for every transcription, with prompt + token usage
- Integration test using a recorded HTTP cassette for Gemini API
**Done when:** user imports an audio file, clicks Transcribe, sees progress, and ends up with a viewable transcript stored in the DB. Resuming a cancelled run picks up where it left off.

### 04 — Tagging UX
**File:** `2026-05-12-04-tagging.md` (written; refine when executing)
**Goal:** Manual tagging end-to-end: select text, apply tag(s), see span in right pane, add memo, play audio range.
**Deliverables:**
- Transcript viewer upgraded with selection handling (text selection within a single segment only)
- Tag popover on selection: codebook search, multi-tag, inline "create new tag" with category picker
- `span_create` / `span_update` / `span_delete` commands; uniqueness of (span, tag) pair via PK
- Right pane: `SpanDetail` showing span text, tags, memo editor
- Audio player: play/pause, seek, speed (0.75/1/1.25/1.5/2×), "Loop span" toggle
- Span markers in transcript viewer (highlight underlay with tag color)
- Overlap visualization for multiple spans on same text
- Keyboard shortcuts: `t` (open tag popover for current selection), `space` (play/pause), `l` (loop span)
**Done when:** user can read transcript, select text, tag it with one or more tags, attach a memo, play back the audio of that span on loop, and see the tag appear in the codebook tree counts.

### 05 — AI features (on-demand)
**File:** `2026-05-12-05-ai-features.md` (written; refine when executing)
**Goal:** Three explicit AI flows with staging UI and cost confirmation. No automatic AI behavior.
**Deliverables:**
- Cost confirmation modal (shows model, est. tokens, est. cost, full prompt collapsible)
- "Generate codebook from interview(s)" — tree-shaped staging UI; accept/reject per item; collision handling
- "Pre-tag whole transcript" — staged spans with dashed underline + AI badge; bulk accept/reject
- "Find more like this" — staged spans for one tag
- Prompt editor in Settings (per-project overrides with reset-to-default)
- `ai_run` audit complete with token usage from API responses
- "Don't ask again this session for this kind of call" preference
- Cassette-based tests for each AI flow
**Done when:** user can run any of the three AI flows from explicit buttons, review proposals in a staging area, and accept/reject without anything mutating the codebook or attaching tags without explicit consent.

### 06 — Import variants
**File:** `2026-05-12-06-import-variants.md` (written; refine when executing)
**Goal:** Non-audio import paths for transcripts.
**Deliverables:**
- "New interview from text" dialog: paste / .txt / .json
- Plain-text parser with `Speaker:` line heuristic (configurable)
- JSON parser for our own segment schema
- Speaker rename UI per interview
- Audio + transcript combo import (timestamp alignment check + approximate flag)
**Done when:** user can start an interview from a pasted transcript or .txt/.json file and tag it normally.

### 07 — Export & stats
**File:** `2026-05-12-07-export.md` (written; refine when executing)
**Goal:** All five export formats from spec §7.6.
**Deliverables:**
- CSV export (per-quote rows)
- Markdown / HTML annotated transcript export
- REFI-QDA XML export (with audio file references)
- Stats report: code frequency by interview/speaker, co-occurrence matrix (CSV + Markdown)
- Codebook export (JSON + CSV)
- Export menu in workspace with scope picker (current interview / whole project / filter by tag)
- Golden-file tests for each format
**Done when:** every export format produces correct, deterministic output verified by golden tests.

### 08 — Polish & packaging
**File:** `2026-05-12-08-polish-packaging.md` (written; refine when executing)
**Goal:** v1.0-ready: settings complete, error states, packaging.
**Deliverables:**
- Settings UI complete (all per-project + global fields, reset-to-default everywhere)
- Czech translations completed for all UI strings
- Error states + retry affordances for failed transcriptions / AI runs
- Recent projects management (rename, remove from list, locate-moved-folder)
- Playwright E2E smoke test against built binary
- Tauri build configuration for Linux (AppImage + deb), macOS (.dmg), Windows (.msi)
- Auto-update strategy decision (built-in updater vs. manual download)
- README with install + usage instructions
- LICENSE decision and file
**Done when:** signed/notarized installers exist for all three platforms, smoke E2E passes on each, README walks a new user through install → first transcription → first tag → first export.

## Cross-cutting standards

These apply to every plan:

- **TDD:** Every feature has a failing test before implementation. Rust unit tests for pure logic, integration tests with cassettes for I/O, Vitest for atom logic and component behavior.
- **Frequent commits:** One logical change per commit. Commit messages follow Conventional Commits (`feat:`, `fix:`, `test:`, `refactor:`, `docs:`).
- **No client-side computation:** Frontend never computes counts, stats, exports, or validation logic. Backend returns ready-to-render data.
- **No automatic AI calls:** Every AI call must originate from an explicit user click. Tested.
- **Uniqueness enforced at DB level:** Codebook constraints rely on SQL `UNIQUE`, not on application logic.
- **i18n from day one:** Any new user-facing string is added to `en.json` and `cs.json` simultaneously. Hard-coded English strings in JSX are forbidden after Plan 01.
- **Plain CSS / CSS Modules:** No CSS-in-JS, no Tailwind. Component styles in `Component.module.css` next to the component.
